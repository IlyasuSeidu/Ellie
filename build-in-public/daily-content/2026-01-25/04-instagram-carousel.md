## Concept

Turn the onboarding trust decision into a saveable framework.

---

## Slides

**Slide 1 — Cover**

If "Tomorrow: Night Shift" is wrong,
the design does not matter.

_Swipe to see what gets built first._

---

**Slide 2 — Real User Tension**

A miner checks their phone before dawn.
One question: what shift am I on tomorrow?

If that answer feels off even once,
they stop trusting the whole app.

Not the screen. Not the button.
The whole thing.

---

**Slide 3 — Wrong Assumption**

The easy assumption:
build something that looks trustworthy.

Polish the welcome screen.
Add smooth animations.
Make the first tap feel premium.

Then wire up the logic underneath.

That order is backwards.

---

**Slide 4 — What Changed**

The schedule answer, the date handling,
and the state layer came first.

The onboarding screens — the animations,
the guided flow, the visual polish —
were built on top of that foundation.

Not before it.

---

**Slide 5 — Proof**

When the onboarding components landed,
they had something solid to sit on.

The progress header feels directed
because the underlying state is directed.

The live pattern preview feels accurate
because the schedule logic was already trustworthy.

The screen reflects the foundation.
It does not replace it.

---

**Slide 6 — The Lesson**

Trust is not a feature you add
after the interface looks good.

It is a structural decision
you make before the first screen appears.

Visible layer + underlying state = one system.
Treat them that way from the start.

---

**Slide 7 — Question**

What do you make trustworthy
before you make it pretty?

Save this if you are building something
where a wrong answer has real consequences.

Drop your answer in the comments.

## Caption

There is a lazy version of this story: build the logic, then build the look.

That is not quite right either.

Building Ellie's onboarding meant making a structural decision early — the schedule answer, the state layer, and the date handling had to be solid before the visible screens made any promises.

When the onboarding components arrived — the guided flow, the animated inputs, the live shift preview — they landed on top of something that was already trustworthy.

That order matters because Ellie will eventually answer one question for a miner before a 5 a.m. alarm:

_What shift am I on tomorrow?_

If the answer underneath is shaky, no welcome screen saves you.

What do you build for trust before you build for appearance?

Built by a miner for shift workers. — Ellie

## Visual Direction

- Cover slide: welcome screen in stone and gold, single declarative line
- Slides 2–4: text-only, one idea per slide, high contrast
- Slide 5: split layout — visible component on one side, a note about what it rests on
- Slide 6: principle line alone, large type, minimal background
- Slide 7: open question centred, save prompt below it
- Keep all slides scannable at a glance — no paragraphs on screen
